# AI Sport 技术方案

## 1. 整体技术架构

AI Sport 的目标不是只在手机上“跑一个模型”，而是完成一条可演示、可交互、可传播的端侧应用链路。围绕这个目标，项目整体分为两条核心技术链：

- `YOLO26n-pose + 本地时序分析链路`
  负责人体关键点检测、动作关键帧选择、视频多帧趋势分析、动作类别判断和次数统计。
- `Qwen3-VL-MNN + 海报内容生成链路`
  负责理解关键帧诊断图和结构化分析结果，生成运动总结、亮点、建议和宣传卡片文案。

在 Android 端，这两条链路最终汇合到同一个 App 流程中：

1. 输入图片或视频
2. 本地完成姿态检测
3. 本地完成关键帧与时序分析
4. 输出诊断图、监测曲线和结构化结果
5. 调用 Qwen-VL-MNN 生成总结与宣传卡片
6. 完成保存与分享

因此，本项目强调的是“模型能力 + 端侧推理 + 产品化输出”的整体落地，而不是单一识别能力展示。

## 2. 模型选型

### 2.1 姿态模型：YOLO26n-pose

姿态检测部分选用 `YOLO26n-pose`，主要原因如下：

- 模型本身具备成熟的人体关键点检测能力，适合深蹲、俯卧撑等运动姿态场景。
- `n` 版本模型参数量更小，更适合在手机端部署和演示。
- 模型支持导出为 `MNN` 格式，能够直接接入 Android 侧 MNN 推理流程。
- 对于比赛场景而言，它在推理速度、工程复杂度和识别效果之间取得了较好的平衡。

在项目中，YOLO-Pose 的职责是“稳定输出关键点”，不直接承担最终动作次数判定。动作次数和动作阶段判断由上层的本地规则分析完成，这样更容易解释，也更便于调试。

### 2.2 视觉理解模型：Qwen3-VL-2B-Instruct-MNN

内容总结部分使用的是 `Qwen3-VL-2B-Instruct-MNN` 这一类 `qwen3-vl-mnn` 资源包。  
从当前工程资源与打包流程看，项目是通过工作区中的 `qwen3-vl-mnn/` 目录同步模型资源，再在 Android 侧通过 MNN LLM runtime 加载。

该模型在本项目中的职责不是动作计数，而是承担更适合视觉语言模型的任务：

- 理解代表帧和诊断图中呈现出的运动状态
- 读取结构化提示信息和本地分析结果
- 输出标题、亮点、建议、标语等总结内容
- 参与宣传卡片文案组织

选择 Qwen3-VL-MNN 的原因主要有四点：

1. 中文图文理解能力较好  
   本项目需要输出适合中文用户直接阅读和传播的总结文案，因此中文理解能力是关键。

2. 适合“图像 + 规则提示”的复合输入场景  
   项目并不是简单地把一张图片丢给模型，而是把 YOLO 诊断图、本地动作结果、运动模式、关键帧标签一起组织成提示输入，让视觉语言模型承担“理解和表述”的工作。

3. 已经具备 MNN 端侧导出版本  
   使用 `Qwen3-VL-2B-Instruct-MNN`，意味着它不是单纯依赖远端推理，而是具备端侧运行基础，更符合比赛要求中的手机端本地部署方向。

4. 2B 等级模型更适合作为手机端展示方案  
   对于当前 Android 手机端演示场景，2B 级多模态模型在效果和可部署性之间更容易取得平衡，便于做出“能跑、能看、能出结果”的完整体验。

### 2.3 模型职责拆分原则

本项目特别强调不同模型职责边界的清晰性：

- `YOLO-Pose` 负责“看见身体”
- `本地规则分析` 负责“看懂动作过程”
- `Qwen-VL-MNN` 负责“把结果说清楚、写漂亮”

这种拆分方式有两个重要好处：

- 比直接让 VL 模型猜动作次数更稳，计数逻辑更可控
- 比只做姿态检测更完整，结果更接近真实产品交付

## 3. 推理框架选型：MNN

本项目端侧推理统一采用 `MNN` 框架。

### 3.1 选择 MNN 的原因

- 对 Android 端部署友好，能较自然地接入 JNI 和本地 so 库
- 同时兼容视觉模型推理和 LLM / VLM 运行时能力
- 适合 CPU 端侧推理，符合本次比赛的移动端本地部署方向
- 可进一步结合 Arm 平台特性进行优化，包括 SME2 相关能力
- 已有较成熟的模型导出和资源组织方式，适合工程打包

### 3.2 在项目中的使用方式

当前工程中，MNN 既承担姿态模型推理，也承担 Qwen-VL 的运行时承载：

- 姿态模型侧：
  `pose_jni.cpp` 中直接使用 `MNN::Interpreter` 和 `MNN::Session` 加载 `yolo26n-pose.mnn`，并通过 CPU 前向推理输出关键点结果。

- Qwen-VL 侧：
  `mnn_jni.cpp` 中通过 `MNN::Transformer::Llm` 相关接口加载 `qwen3-vl-mnn` 资源包，对结构化提示和视觉上下文进行推理。

也就是说，项目不是“视觉模型用一种方案，多模态模型再用另一种方案”，而是尽量统一在 MNN 这一套端侧框架之上，降低整体工程复杂度。

## 4. Qwen3-VL-MNN 端侧落地方案

### 4.1 资源组织方式

从工程构建流程来看，`app/build.gradle.kts` 中会将工作区下的 `qwen3-vl-mnn/` 自动同步到 Android 资产目录：

- 工作区源目录：`qwen3-vl-mnn`
- 生成资产目录：`build/generated/ai_sport_assets/qwen3-vl-mnn`
- App 启动后通过 `mnnEngine.loadModelBundle("qwen3-vl-mnn")` 加载

这种做法的优点是：

- 模型资源和 App 代码解耦
- 便于替换、升级不同版本的 VL 模型包
- 比手动散落放置模型文件更稳定

### 4.2 模型作用链路

Qwen3-VL-MNN 在本项目中不直接读取原始运动视频全序列，而是读取以下“已经被整理过的输入”：

- 代表帧图片
- YOLO 姿态诊断图
- 本地运动分析结果
- 关键动作标签
- 置信度、动作模式等结构化提示

这种设计思路很重要。  
因为端侧多模态模型资源有限，如果让它直接承担“从视频原始帧中自己推理动作与次数”的全部任务，代价太高，且结果不稳定。  
而通过前置姿态分析先把任务压缩为“理解结果并生成文案”，可以大幅降低模型负担，提升整体可用性。

### 4.3 端侧多模态模型使用原则

项目对 Qwen3-VL-MNN 的使用遵循三个原则：

1. 只让模型做擅长的事  
   即看图理解、总结、文案组织，而不是做严苛的时序计数。

2. 先做结构化，再做生成  
   先由本地分析模块产出动作类别、代表帧、曲线趋势，再交给 VL 模型生成自然语言结果。

3. 以结果交付为目标，而不是追求最大模型链路  
   本项目要交付的是一款手机应用，不是最大化地把所有能力都塞给一个模型。

## 5. 手机端推理与 SME2 适配思路

### 5.1 目标设备与平台背景

项目的目标展示设备是高性能 Android 手机，采用 Armv9.3-A 架构，并开启 `SME2` 能力。  
在比赛语境下，这一点非常关键，因为它意味着本项目不是泛泛地说“模型在手机上能跑”，而是明确围绕 Arm 新指令集能力做端侧推理落地。

### 5.2 什么是 SME2

`SME2` 可以理解为 Arm 新一代面向矩阵运算、向量计算和高性能数值处理的扩展能力。  
对于端侧 AI 推理来说，它的重要意义在于：

- 能提升矩阵相关运算效率
- 对长 prompt、多层 attention、权重量化解压等场景更有帮助
- 能在 CPU 本地推理链路中带来真实可测的优化空间

对于 Qwen3-VL-MNN 这类多模态模型，以及 MNN LLM runtime 的 prefill / decode 过程来说，SME2 的价值不仅体现在“是否能运行”，更体现在“能否在手机端把复杂推理做得更可用”。

### 5.3 当前工程中的 SME2 组织方式

从当前工程可见，项目已经明确保留了两套 MNN Android 构建目录：

- `build-android-arm64-4b-sme2-on`
- `build-android-arm64-4b-sme2-off`

并通过 `app/build.gradle.kts` 中的 `mnnBuildDir`/`MNN_BUILD_DIR` 切换实际打包的运行时 so：

- `libMNN.so`
- `libMNN_Express.so`
- `libllm.so`

这意味着项目不是口头描述“开启了 SME2”，而是在工程层面具备：

- 可切换 ON/OFF 的两套 runtime
- 同一模型、同一 App、不同 runtime 的对比基础
- 可用于答辩和性能说明的实验条件

### 5.4 Qwen-VL-MNN 与 SME2 的关系

对于 Qwen3-VL-MNN 这类端侧多模态模型，SME2 的意义主要体现在以下几个方面：

1. 降低长上下文 prefill 压力  
   多模态输入往往不是一句短文本，而是包含图像理解提示、结构化结果和总结指令，这会让 prefill 阶段开销上升。SME2 能帮助优化这部分 CPU 推理过程。

2. 提升端侧生成响应效率  
   在宣传卡片文案生成、运动总结输出等环节，decode 速度直接影响用户等待时间。SME2 在较理想温控条件下有机会带来更好的响应效率。

3. 提高手机端产品可用性  
   对比赛项目而言，真正重要的不只是 benchmark 分数，而是“用户点击生成海报后，要等多久”。SME2 的价值最终会体现在端到端等待时间改善上。

### 5.5 运行时编译与配置思路

从当前工程资源中可以看到，MNN Android 构建已围绕以下方向组织：

- `MNN_BUILD_LLM=ON`
- `MNN_LOW_MEMORY=ON`
- `MNN_SUPPORT_TRANSFORMER_FUSE=ON`
- `MNN_ARM82=ON`
- `MNN_SME2=ON`
- `MNN_CPU_WEIGHT_DEQUANT_GEMM=ON`

这些配置组合说明，项目对端侧多模态/LLM 推理的考虑不是“能编就行”，而是围绕：

- 低内存
- CPU 权重量化推理
- Transformer 相关优化
- Arm 指令集能力利用

进行构建的。

## 6. Android 端侧适配思路

### 6.1 模型与资源打包

项目中的姿态模型和 Qwen-VL-MNN 资源都采用“构建前同步到 generated assets”的方式管理，而不是运行时临时下载或路径拼接。

这样做的好处是：

- 资源路径统一
- 构建结果更可复现
- 更适合比赛现场演示
- 避免模型缺失、路径错误等低级问题

### 6.2 JNI 与本地推理桥接

为了在 Android App 中稳定接入 MNN，本项目使用 `Kotlin + JNI + C++` 组合：

- Kotlin 负责 UI、流程控制和结果展示
- C++ 负责 MNN runtime 调用和模型前向推理
- JNI 作为中间桥梁传递输入输出

这使得姿态模型和 Qwen-VL 模型都可以在统一工程中被组织起来，而不是零散依赖外部脚本。

### 6.3 图片与视频输入适配

应用支持：

- 相册选图
- 相机拍照
- 相册选视频
- 相机录制视频

其中视频模式下会进行：

- 多帧采样
- 关键帧优选
- 代表帧选择
- 采样预览图生成
- 曲线可视化

这部分保证了输入不是“只能处理单张测试图”，而是能适配真实用户的运动录制场景。

### 6.4 本地动作分析层

YOLO-Pose 只负责关键点输出，本项目额外在 Android 端实现了本地动作分析层，用于：

- 肘角、膝角等几何特征计算
- 曲线信号构建
- 状态机与波形结合的次数判断
- 动作模式约束

这样做的优势在于：

- 延迟低
- 不必额外训练动作计数模型
- 可解释性强
- 更适合比赛答辩展示工程思路

### 6.5 结果可视化与分享

项目不是直接输出 JSON，而是做了完整的端侧结果展示：

- 代表帧
- YOLO 诊断图
- 采样帧诊断图
- 视频监测曲线
- 宣传卡片
- 保存与分享

这一步让整个系统从“识别工具”变成“内容产品原型”，也让 Qwen-VL-MNN 的文案生成价值得到真正发挥。

## 7. 技术方案总结

AI Sport 的技术方案可以概括为一句话：

**用 YOLO-Pose 负责看懂人体，用本地时序分析负责看懂动作过程，用 Qwen3-VL-MNN 负责把结果表达成用户愿意分享的内容，并通过 MNN + SME2 路线把整套能力压到手机端。**

更具体地说：

- `YOLO26n-pose` 解决姿态关键点检测
- `本地规则与时序分析` 解决动作类别和次数问题
- `Qwen3-VL-2B-Instruct-MNN` 解决图文理解与海报文案生成问题
- `MNN` 统一承载端侧推理
- `SME2 ON/OFF 双 runtime` 提供了面向比赛要求的工程化端侧优化基础
- `Android + JNI + 资产同步 + 分享导出` 负责把模型能力变成实际产品能力

因此，本项目并不是一个单点模型 Demo，而是一套围绕“手机上的创意 AI”方向设计的、具有完整工程链路的端侧产品方案。

## 8. 后续 ToDo 方案

当前版本已经完成了“姿态检测 + 本地规则分析 + 视觉总结 + 宣传卡片生成”的闭环，但从产品持续演进和技术深化角度来看，后续仍有明确的升级路线。

### 8.1 关键帧与多帧数据集建设

目前项目中的动作计数主要依赖 `YOLO-Pose` 输出的关键点，再结合本地状态机和波形逻辑做分析。这种方案优点是轻量、可解释、上线快，但对于复杂姿态、不同拍摄角度和遮挡场景，仍然存在稳定性边界。

因此，后续计划围绕真实运动视频持续建设数据资产，重点沉淀两类数据：

- `关键帧数据`
  包含代表帧、底部帧、顶点帧、异常帧、误识别帧等，用于后续单帧姿态质量判断、动作阶段分类和错误动作识别。
- `连续时序数据`
  包含多帧关键点序列、动作波形曲线、标签化次数信息，用于后续训练更稳健的动作计数模型。

数据建设重点包括：

- 俯卧撑、深蹲等核心动作的多角度视频采集
- 不同距离、不同背景、不同体型用户的数据覆盖
- 标准动作与非标准动作的标签区分
- 关键阶段帧的人工校验和标注

### 8.2 从规则计数升级到深度学习计数

当前版本中，动作次数主要通过“关键点角度 + 状态机 + 自适应波形”完成，这适合早期快速落地。  
后续计划进一步引入深度学习方法，对动作相关关键帧和关键点时序进行训练，从而提升复杂场景下的计数准确率和泛化能力。

后续可以尝试的方向包括：

1. `关键点时序分类模型`
   将连续关键点序列输入轻量级时序模型，对动作阶段进行分类，例如“上升阶段 / 底部阶段 / 顶部阶段”，再基于阶段切换完成更稳的计数。

2. `关键帧阶段识别模型`
   针对代表帧、底部帧、顶点帧进行监督训练，让模型学习判断当前帧属于哪个动作阶段，从而辅助状态机计数。

3. `时序检测 / 重复动作计数模型`
   进一步探索使用轻量级时序网络，对整段动作视频直接做重复周期检测，降低对手工阈值的依赖。

4. `动作质量与风险分析模型`
   在“计数正确”之外，继续训练动作质量分类模型，用于识别膝内扣、下蹲不足、俯卧撑塌腰等问题。

在工程路径上，推荐采取“分阶段演进”方式：

- 第一阶段：继续保留规则计数，增强数据采集与标注
- 第二阶段：训练动作阶段识别模型，作为规则系统的辅助判别器
- 第三阶段：训练完整时序计数模型，在规则方案之上逐步替换或融合

这样做的好处是不会推翻当前可运行版本，而是在现有成果上持续演进。

### 8.3 更多动作类型扩展

当前版本重点验证俯卧撑和深蹲两类动作。后续计划逐步扩展到：

- 开合跳
- 弓步蹲
- 平板支撑
- 仰卧起坐
- 高抬腿

扩展策略不是简单增加标签，而是围绕“关键点特征是否清晰、计数逻辑是否稳定、是否适合端侧展示”来做动作优先级选择。

### 8.4 Qwen-VL 结果解释与个性化输出增强

目前 Qwen3-VL-MNN 主要承担总结与海报文案生成。后续还可以进一步增强：

- 根据不同运动类型生成差异化宣传卡片模板
- 对动作质量问题生成更细化的中文解释
- 将本地时序分析结果转化为更自然的训练建议
- 结合用户历史记录，生成个性化运动报告

这一步会让 Qwen-VL 从“文案生成器”进一步升级为“运动内容理解与表达引擎”。

### 8.5 开源计划

本项目后续计划逐步开源，目标不是只开放一个静态页面，而是开放完整的端侧 AI 应用原型链路，方便开发者学习和复现。

计划开源的内容包括：

- Android App 主工程
- YOLO-Pose MNN 接入流程
- Qwen3-VL-MNN 资源组织与加载方式
- 姿态分析、关键帧选择、波形计数等核心逻辑
- 海报生成与分享模块
- 示例文档与部署说明

在条件允许的情况下，还计划进一步开放：

- 关键帧标注规范
- 动作时序训练数据格式定义
- 部分去隐私化的样例数据
- 训练与评估脚本

开源的价值主要体现在三点：

1. 方便更多开发者复现“手机上的创意 AI”方案
2. 形成围绕端侧运动 AI 的可持续协作基础
3. 为后续动作计数模型训练和社区共建提供数据与工程支撑

### 8.6 后续路线总结

后续 ToDo 方案的核心思路，不是推翻当前版本，而是在现有闭环基础上继续升级：

- 短期：完善规则分析、提升计数稳定性、补齐分享体验
- 中期：建设关键帧与时序数据集，引入深度学习计数模型
- 长期：逐步开源，扩展动作类型，形成完整的端侧运动 AI 应用框架

这也意味着 AI Sport 不只是一个比赛 Demo，而是有机会持续演化为一个真正可扩展、可研究、可开源的手机端运动 AI 项目。
