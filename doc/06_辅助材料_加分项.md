## 2. 关键工程文件说明

### 2.1 Android 主界面与交互流程

关键文件：

- `app/src/main/java/com/aisport/ui/MainActivity.kt`

作用说明：

- 负责图片、视频、相机输入流程
- 管理运动分析模式切换
- 串联 YOLO-Pose 分析、Qwen-VL 总结、海报生成和分享流程
- 承担主要 UI 展示，包括代表帧、诊断图、视频曲线和宣传卡片

这一文件体现了项目不是单个模型脚本，而是一个完整 Android App 的用户交互入口。

### 2.2 YOLO-Pose 端侧接入

关键文件：

- `app/src/main/java/com/aisport/pose/NativePoseEstimator.kt`
- `app/src/main/cpp/pose_jni.cpp`

作用说明：

- Kotlin 层定义姿态推理接口与结果结构
- C++ 层通过 MNN `Interpreter + Session` 完成 `yolo26n-pose.mnn` 的本地加载与推理
- 将关键点、骨架边、置信度等信息返回上层业务逻辑

这一部分是项目中“手机端真实姿态推理”的核心落地点。

### 2.3 本地动作分析与计数逻辑

关键文件：

- `app/src/main/java/com/aisport/exercise/ExerciseAnalyzer.kt`
- `app/src/main/java/com/aisport/video/VideoFrameSampler.kt`

作用说明：

- 对视频采样帧进行关键点特征提取
- 计算肘角、膝角、身体线性度、躯干水平度等几何特征
- 使用状态机与自适应波形逻辑完成俯卧撑、深蹲等动作的本地次数统计
- 生成视频时序曲线与代表帧候选

这一部分体现了项目当前不是把动作计数交给黑盒模型，而是采用了可解释、可调试、可持续升级的端侧分析策略。

### 2.4 Qwen3-VL-MNN 总结链路

关键文件：

- `app/src/main/java/com/aisport/vision/SportAnalyzer.kt`
- `app/src/main/java/com/aisport/engine/MnnEngine.kt`
- `app/src/main/cpp/mnn_jni.cpp`

作用说明：

- 加载 `qwen3-vl-mnn` 模型资源
- 组织图像与结构化提示词输入
- 输出运动总结、建议、标题和宣传语
- 将多模态推理结果封装为应用可直接消费的结构化对象

这一部分体现了 Qwen-VL-MNN 在项目中的真实职责：  
**不是负责动作次数，而是负责总结与内容表达。**

### 2.5 海报生成与分享模块

关键文件：

- `app/src/main/java/com/aisport/poster/PosterComposer.kt`
- `app/src/main/java/com/aisport/poster/PoseDebugRenderer.kt`
- `app/src/main/java/com/aisport/poster/WorkoutTimelineRenderer.kt`

作用说明：

- 绘制 YOLO 姿态诊断图
- 生成运动曲线图
- 组合代表帧、文字总结和版式元素，生成宣传卡片
- 支持保存到相册与系统分享 / 微信分享

这一部分体现了项目从“分析结果”进一步走向“内容产出”的产品化能力。

### 2.6 构建与端侧资源同步

关键文件：

- `app/build.gradle.kts`
- `app/src/main/cpp/CMakeLists.txt`

作用说明：

- 自动同步 `qwen3-vl-mnn` 资源到 Android 资产目录
- 自动同步 `yolo26n-pose.mnn` 到生成资源目录
- 自动选择和打包 MNN runtime so
- 支持通过 `mnnBuildDir` 在 `sme2-on / sme2-off` 之间切换

这一部分是比赛中非常重要的工程加分项，因为它证明项目不仅依赖模型文件，还真正具备端侧构建组织能力。

## 3. 关键技术难点与解决方案

### 3.1 技术难点一：YOLO-Pose 在手机端落地

难点说明：

- 姿态模型在 PC 侧跑通并不等于 Android 端能稳定运行
- 需要解决模型导出、资源打包、JNI 接口、输入输出张量适配等问题

解决方案：

- 选用更适合端侧部署的 `YOLO26n-pose`
- 提前导出为 `MNN` 格式，避免移动端运行时转换
- 在 C++ 侧通过 MNN 原生接口加载模型，减少中间层损耗
- 在 Kotlin 层封装稳定接口，便于上层业务复用

### 3.2 技术难点二：动作计数不能只靠单帧识别

难点说明：

- YOLO-Pose 输出的是关键点，不是直接的“俯卧撑 10 次”
- 单帧判断很难完成稳定计数
- 用户动作节奏、角度、遮挡、机位差异都会影响结果

解决方案：

- 对视频做多帧采样，而不是只取单张图
- 在端侧构建几何特征和趋势曲线
- 引入状态机与自适应波形双策略，提高计数鲁棒性
- 保留可解释逻辑，为后续训练时序模型做过渡

### 3.3 技术难点三：Qwen-VL 的职责边界设计

难点说明：

- 如果直接让视觉语言模型负责动作识别与次数统计，结果容易漂移
- 端侧多模态模型资源有限，不适合承担全量时序分析任务

解决方案：

- 将动作识别与动作计数留在本地分析层
- 将 Qwen3-VL-MNN 专门用于“看图总结、内容组织、宣传文案生成”
- 采用“先结构化分析，再自然语言生成”的链路，降低模型负担

这种分工使得整套系统既保持稳定性，也保留了生成式 AI 的表达能力。


### 3.5 技术难点四：比赛要求中的 SME2 适配

难点说明：

- 比赛强调手机端本地推理与 Arm SME2 能力利用
- 这要求项目不能只说“支持端侧”，而要具备切换和验证基础

解决方案：

- 保留 `build-android-arm64-4b-sme2-on` 与 `build-android-arm64-4b-sme2-off` 两套 MNN runtime
- Gradle 构建阶段通过 `mnnBuildDir` 切换打包
- 统一同步 `libMNN.so / libMNN_Express.so / libllm.so`
- 让 Qwen-VL-MNN 的运行时具备可验证的端侧优化路径

## 4. 工程完成度补充说明

本项目当前版本已经具备以下工程完成度特征：

- 不是网页演示，而是 Android 原生 App
- 不是单纯云端调用，而是包含真实端侧推理链路
- 不是只有模型输出，而是包含结果展示、海报生成和分享流程
- 不是只验证姿态模型，而是同时验证了视觉理解模型与产品表达能力

这一点使得项目在参赛材料中更接近“可运行产品原型”，而不是算法片段拼接。

## 5. 当前版本边界说明

为了保证材料真实可信，也需要说明当前版本边界：

- 俯卧撑、深蹲计数逻辑仍在持续调优
- 当前计数主要基于规则分析与时序信号，尚未引入训练好的动作时序模型
- Qwen3-VL-MNN 负责总结与文案，不直接承担动作计数
- 当前重点是形成完整闭环，后续再继续提升精度与动作覆盖范围

这类边界说明不是减分项，反而能体现项目对真实工程状态的诚实表达。
